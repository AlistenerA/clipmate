# V0.5_X_ARCHITECTURE.md — ClipMate v0.5.x 架构底座

> v0.5.x 用于承接 v0.5.0 发布后的架构级优化。每完成一个可验证功能点，patch 版本递增 0.0.1。

---

## v0.5.1 范围

v0.5.1 不改变用户可见功能，不新增权限，不新增依赖。目标是把现有剪藏链路拆出稳定边界，方便后续继续吸收 Save to Notion 参考项目中的会话、目标选择和媒体保存思路。

---

## 分层边界

### `src/features/capture`

负责把 `ExtractedContent + tags + note` 组装为 `ClipDraft`。

- `createClipDraft()`：统一草稿创建入口
- `getDraftBodyText()`：统一正文文本读取逻辑，优先 markdown，回退 contentText
- `isDraftSaveable()`：统一可保存判断

### `src/features/session`

负责描述一次剪藏保存会话。

- `createClipSession()`：创建 session，包含草稿、目标、状态和时间戳
- `markClipSessionSaving/Saved/Failed()`：纯函数状态转换
- `createSaveToNotionPayloadFromSession()`：把 session 转换为现有 background 保存消息

当前 v0.5.1 只把 session 用于 popup 保存入口，不持久化 session，也不替换现有 hook 的 UI 状态。

### `src/features/notion`

负责把保存请求归一化为 Notion 保存计划。

- `createNotionSavePlan()`：集中处理 token/page/content 校验、blocks 构建、history retry/update 元数据
- `handleSaveToNotion()` 保留 IO 编排：读 settings、调用 Notion API、写 history

---

## 参考 Save to Notion 的取舍

采用：

- 会话式保存思路：一次保存由 draft + target + status 组成
- 保存前计划生成：先校验和构建 blocks，再执行外部 IO
- 目标元数据明确传递：targetId / targetName / pageId 随 session 进入保存请求

不采用：

- 不引入非官方 Notion 私有接口或 cookie 依赖
- 不扩大 manifest 权限
- 不把 DOM 提取、UI 状态、Notion IO 混进同一个模块
- 不整段复制 third-party 实现

---

## 后续接入方向

- 把 History retry、popup 保存、未来 image upload 都统一到 `ClipSession` 生命周期
- 将 Notion target resolution 独立成 `features/targets`
- 如后续接入 Notion File Upload，先在 `features/notion` 增加 plan 层策略，再评估权限和隐私边界
