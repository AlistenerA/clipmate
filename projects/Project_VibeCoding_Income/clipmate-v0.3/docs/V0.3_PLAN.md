## 一、v0.3 当前定位

ClipMate v0.3 是 v0.2 之后的首个功能增强大版本。v0.2 已完成全部编码工作（12 个 Session，321 tests），v0.3 的目标是在不破坏 v0.2 稳定基线的前提下，选取一条唯一主线方向进行升级。

**核心原则**：
- 不一次性实现 AI、多平台、OCR、付费
- 不在第一条主线未完成前分叉第二条主线
- 不新增不必要的权限
- 新增外部 API 必须默认关闭，由用户主动开启

## 三、v0.3 候选方向

### 方向 A：AI 摘要与 AI 标签

**可能功能**：
- 用户自带 API Key（支持 OpenAI / Claude / DeepSeek / 智谱等主流模型）
- 对剪藏正文生成 1-3 句中文摘要
- 根据正文推荐 3-5 个标签
- 保存到 Notion 时可选择是否包含摘要（摘要作为 callout block 或独立段落）
- 历史记录中保存 AI 摘要结果
- Options 中管理 AI 配置（API Key、模型选择、开关）

**影响评估**：

| 维度 | 评估 |
|------|------|
| 是否新增权限 | **否**（API Key 由用户输入，调用走现有 host_permissions 或新增一处 host） |
| 是否新增外部 API | **是** — 需要接入至少一个 LLM API（用户自带 Key） |
| 是否传输用户内容 | **是** — 正文会发送到模型 API；必须在 UI 中明确告知用户 |
| 是否需要更新隐私政策 | **是** — 需声明正文可能传输到第三方 LLM API，以及用户可关闭此功能 |
| 是否需要更新商店文案 | **是** — 需增加 AI 功能介绍和隐私说明 |
| 是否需要新增配置 | **是** — API Key、模型选择、摘要开关、标签开关 |
| 是否影响 Notion 保存链路 | **是** — 保存时需要决定摘要和标签是否附加到 Notion page |
| 是否需要数据迁移 | **是** — settings 需新增 AI 配置字段；历史记录需新增 aiSummary / aiTags 字段 |
| 审核风险 | **中** — AI 功能需在商店审核中提供清晰的隐私说明和使用流程 |

**风险**：
- 用户 API Key 泄露风险（需 `type="password"` 输入框、不打印 Key 到日志）
- 费用（由用户承担 API 调用费用）
- 正文隐私（明确的用户告知和默认关闭策略）
- 模型不可用时的 fallback 策略

**建议**：
- 优先做"用户自带 Key + 明确开关 + 默认关闭"
- 不要在 v0.3 首轮就接多个模型供应商（先做一个，后续扩展）
- 不要默认上传全文，必须由用户主动触发（Popup 中加一个按钮）
- API Key 仅存储在 chrome.storage.local，不传输给插件维护者

### 方向 B：Notion Database 属性映射

**可能功能**：
- 保存到 Notion database（而非普通 page）
- 支持 title / url / tags / date / source / status 等属性映射
- 用户在 Options 中配置 database id 和字段名
- 可能与方向 A 组合（摘要写入属性）

**影响评估**：

| 维度 | 评估 |
|------|------|
| 是否新增权限 | **否** |
| 是否新增外部 API | **否**（仍调用 Notion API） |
| 是否传输用户内容 | **否**（与 v0.2 相同） |
| 是否需要更新隐私政策 | **是** — 需描述 database 保存行为 |
| 是否需要更新商店文案 | **是** |
| 是否需要新增配置 | **是** — database id、属性映射字段 |
| 是否影响 Notion 保存链路 | **是** — blocks.ts 和 client.ts 需大幅改造 |
| 是否需要数据迁移 | **是** — targets 结构需区分 page target 和 database target |
| 审核风险 | **低** — 属于功能增强，不涉及新权限 |

**风险**：
- 配置复杂度高，UI 设计容易变重
- Notion API 中 database 属性的创建/查询/更新错误场景多
- 用户不理解 schema 映射概念

**建议**：
- 可作为 v0.3 或 v0.3+ 的独立子版本评估
- 必须先做只读方案设计，不直接实现
- 建议先做完方向 A 再做方向 B，避免两条线交叉

---

### 方向 C：Obsidian / 本地 Markdown 导出

**可能功能**：
- 一键将剪藏内容导出为本地 Markdown 文件
- 支持 Obsidian 风格 frontmatter（YAML 头部）
- 自定义文件命名模板（日期 + 标题）
- 下载到本地（使用 `chrome.downloads` API 或复制到剪贴板）

**影响评估**：

| 维度 | 评估 |
|------|------|
| 是否新增权限 | **可能** — `downloads` 权限（如下载到磁盘）；如仅复制到剪贴板则无需 |
| 是否新增外部 API | **否** |
| 是否传输用户内容 | **否** |
| 是否需要更新隐私政策 | **否**（如无需新权限且不调用外部 API） |
| 是否需要更新商店文案 | **是** |
| 是否需要新增配置 | **是** — 导出格式、文件命名模板 |
| 是否影响 Notion 保存链路 | **否** — 独立链路 |
| 是否需要数据迁移 | **是** — settings 新增导出配置 |
| 审核风险 | **低** — 仅新增 `downloads` 权限需说明 |

**风险**：
- `downloads` 权限在 Edge Add-ons 审核中需要合理说明
- 如果只用剪贴板方案，体验不如直接下载

**建议**：
- 是隐私和 API 风险最低的方向
- 适合作为"如果用户不接受外部 API"的替代方案
- 可以先做剪贴板方案，后续迭代加下载

---

### 方向 D：多平台保存（飞书 / 语雀）

**可能功能**：
- 新增飞书文档保存
- 新增语雀知识库保存
- 平台切换 UI

**影响评估**：

| 维度 | 评估 |
|------|------|
| 是否新增权限 | **是** — 需新增 host_permissions（飞书 API / 语雀 API） |
| 是否新增外部 API | **是** — 飞书 Open API、语雀 API |
| 是否传输用户内容 | **是** — 内容发送到平台 API |
| 是否需要更新隐私政策 | **是** — 需说明多平台数据流 |
| 是否需要更新商店文案 | **是** |
| 是否需要新增配置 | **是** — 每个平台需独立鉴权和配置 |
| 是否影响 Notion 保存链路 | **是** — 需抽象平台适配器接口 |
| 是否需要数据迁移 | **是** — settings 架构变更 |
| 审核风险 | **高** — 多平台鉴权、权限膨胀、审核周期长 |

**建议**：
- **不建议 v0.3 做**
- 复杂度极高，每个平台需独立开发、测试、维护
- 最早 v0.4+ 再评估

---

### 方向 E：OCR / 截图回退

**可能功能**：
- 对无法提取正文的网站使用截图 + OCR
- 保存网页视觉内容

**影响评估**：

| 维度 | 评估 |
|------|------|
| 是否新增权限 | **是** — 可能需 `<all_urls>` 截图权限、activeTab 升级 |
| 是否新增外部 API | **是** — OCR API（或本地 OCR 引擎） |
| 是否传输用户内容 | **是** — 截图/文本发送到 OCR API |
| 是否需要更新隐私政策 | **是** |
| 是否需要更新商店文案 | **是** |
| 是否需要新增配置 | **是** — OCR 服务配置 |
| 是否影响 Notion 保存链路 | **否**（独立链路） |
| 是否需要数据迁移 | **是** |
| 审核风险 | **高** — 截图和 OCR 权限敏感 |

**建议**：
- **不建议 v0.3 做**
- 除非用户明确把 OCR 作为核心方向，否则暂缓

## 四、推荐优先级

推荐优先顺序：

| 优先级 | 方向 | 理由 |
|:---:|------|------|
| **1（推荐）** | **方向 A：AI 摘要与 AI 标签** | 用户价值最高，差异化竞争点，技术可行，审核风险可控 |
| 2（备选） | 方向 B：Notion Database 属性映射 | 增强 Notion 集成深度，适合重度 Notion 用户 |
| 3（备选） | 方向 C：Obsidian / 本地 Markdown 导出 | 隐私风险最低，适合不接受外部 API 的用户 |
| 暂缓 | 方向 D：多平台保存 | 复杂度过高 |
| 暂缓 | 方向 E：OCR / 截图回退 | 审核风险高 |
| 暂缓 | 方向 F：付费 / License / 后端 | 不在当前阶段 |

**当前建议**：优先评估 AI 摘要与 AI 标签，但必须由用户确认是否接受外部 API、隐私政策更新和用户自带 API Key 模式。若用户不接受外部 API，则优先考虑 Obsidian / 本地 Markdown 导出。

---

## 五、不推荐同时做的内容

以下组合**禁止**出现在同一个 v0.3 版本中：

1. AI + 多平台 → 开发量和测试量成倍增长
2. AI + OCR → 两条外部 API 链互相干扰
3. AI + Database → 两条方向都改动 Notion 保存链路，冲突风险高
4. AI + 付费 → 需要后端的付费系统与前端 AI 功能没有直接关系

v0.3 应选取**唯一一条主线**，在主线完成并通过人工 QA 后才考虑扩展。

## 八、风险清单

| 编号 | 风险 | 严重度 | 缓解措施 |
|:---:|------|:---:|------|
| R01 | AI API Key 泄露 | 🔴 高 | `type="password"` 输入框，日志不打印 Key，仅存本地 |
| R02 | 用户不知情下正文上传到 LLM API | 🔴 高 | 默认关闭，UI 明确告知，用户主动触发 |
| R03 | LLM API 不可用时用户困惑 | 🟡 中 | 错误映射，友好提示，允许用户关闭 AI 后正常剪藏 |
| R04 | settings 迁移 v2→v3 破坏旧配置 | 🟡 中 | 深拷贝后原子写回，迁移仅在首次检测低版本时执行 |
| R05 | AI 摘要/标签结果不理想 | 🟡 中 | 使用 system prompt 优化质量，允许用户编辑结果 |
| R06 | 历史记录中 AI 结果字段膨胀 | 🟡 中 | 可选的 aiSummary/aiTags 字段，用户可清除历史 |
| R07 | v0.3 范围膨胀（想做多个方向） | 🟡 中 | S0 明确唯一主线，后续 Session 严格遵循 |
| R08 | manifest 版本号提前修改导致半成品混淆 | 🟡 中 | release 文档/打包 Session 才改版本号 |

## 九、每个 Session 的交付标准（继承 WORKFLOW_RULES.md）

1. `npm run lint` — 0 errors, 0 warnings
2. `npm run test` — 全部通过
3. `npm run build` — 构建成功
4. 更新交接文档（CURRENT_STATUS / CHANGELOG_AGENT / TEST_LOG / ISSUES / DECISIONS）
5. 不提交 dist/、zip、node_modules/、.env、工具目录
6. 不输出真实 Token、正文全文、备注全文、API Key
7. 发给 ChatGPT 审查后再决定是否 commit
